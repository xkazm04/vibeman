2. **conductorOrchestrator.ts** -- Add triage checkpoint flow after `executeTriageStage()` returns scored items:

   a. Add `waitForResumeWithTimeout(runId, timeoutMs)` function:
      - Same polling loop as existing `waitForResume` (2000ms interval)
      - Track elapsed time; if exceeds `timeoutMs`, return `'timeout'`
      - If run status changes from 'paused', return `'resumed'`
      - If abort detected, return `'aborted'`
      - Use `(context.status as string)` assertion after the wait (TS narrowing gotcha)

   b. Wire triage checkpoint in orchestrator (after triage scoring, before batch/spec):
      - Read `checkpointConfig.skipTriage` (default false if absent -- use `?? false`)
      - If NOT skipping:
        - Import and call `detectBrainConflicts(scoredItems, projectId)` from `src/lib/brain/conflictDetector`
        - Build `TriageCheckpointData` with items, conflict results, timeoutAt (now + 3600000ms), createdAt
        - Write to DB: `updateRunInDb(runId, { checkpoint_type: 'triage', triage_data: JSON.stringify(triageData) })`
        - Set status to 'paused': `conductorRepository.updateRunStatus(runId, 'paused')`
        - Log: `log('triage', 'info', 'Triage checkpoint -- waiting for user review')`
        - `const result = await waitForResumeWithTimeout(runId, 3_600_000)`
        - If result === 'timeout': set status='interrupted', log reason 'Triage checkpoint timed out after 1 hour', clear triage_data, return
        - If result === 'aborted': set status='interrupted', return
        - After resume: read triage_data from DB, parse decisions array, apply approve/reject via tinder API for each item
        - Clear checkpoint: `updateRunInDb(runId, { checkpoint_type: null })`
        - Set status back to 'running'
      - If skipping: log 'Triage bypassed (skipTriage=true)', auto-approve all items via tinder API, continue
