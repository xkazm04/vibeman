  <action>
Create `goalAnalyzer.ts` exporting `executeGoalAnalysis(input: GoalAnalyzerInput): Promise<GoalAnalyzerOutput>`.

**Structure:**

1. **File discovery**: Call `discoverRelevantFiles()` with goal's contextId (parse from target_paths if it contains a context reference, or null) and goal description text.

2. **Brain context injection**: If `goal.use_brain` is truthy, call `getBehavioralContext(input.projectId)`. If `hasData` is true, build a Brain section for the prompt using topInsights and patterns. Format as "This project has learned..." institutional knowledge (see RESEARCH.md buildBrainSection example). If hasData is false or use_brain is falsy, set Brain section to empty string.

3. **Context list for domain assignment**: Query project contexts from contextDb to get available context names and IDs. Format as a simple list for the LLM to select from when assigning domains.

4. **Build analysis prompt**: Create a single comprehensive prompt with these sections:
   - **Goal**: The user's goal title and description
   - **Brain Patterns** (conditional): The formatted Brain section
   - **Codebase Context**: File tree for breadth, then full file contents for the discovered files
   - **Available Domains**: Context list for domain assignment
   - **Analysis Perspectives**: Three scan type perspectives woven as lenses:
     - zen_architect: Look for architectural gaps, missing abstractions, structural debt
     - bug_hunter: Look for error handling gaps, edge cases, fragile patterns
     - ui_perfectionist: Look for UX gaps, accessibility issues, inconsistent patterns
   - **Output Format**: Request structured JSON with `{ gaps: GapItem[], backlogItems: BacklogItemInput[] }`. Specify the exact schema including all fields. Emphasize effort/impact/risk as 1-10 scale, source tagging (structural_analysis vs creative_suggestion), relevanceScore as 0-1.
   - Instruct the LLM to only identify gaps relevant to the stated goal, not generic improvements.
   - Brain data should guide what gaps are found (analysis guidance per CONTEXT.md), but NOT influence effort/impact/risk scoring.

5. **LLM call**: Use the `/api/ai/chat` proxy pattern from diffReviewer.ts. Use `config.scanModel || 'sonnet'` for model selection.

6. **Response parsing**: Extract JSON from response using regex pattern `responseText.match(/\{[\s\S]*\}/)`. Parse and validate the result. Handle common edge cases: code fences, malformed JSON. If parsing fails, return empty gap report and empty backlog.

7. **Build output**: Construct `GoalAnalyzerOutput` with:
   - `gapReport`: Populate with goalId, analyzedFiles (from discovery), gaps (from LLM response), analyzedAt timestamp
   - `backlogItems`: Map from LLM response, validate effort/impact/risk are 1-10, validate contextId exists in project contexts (fallback to null if not), ensure source field is set

**Error handling**: If LLM call fails or returns non-200, log warning and return empty output (non-blocking, matches Phase 4 LLM failure pattern). Do NOT throw -- let the orchestrator handle gracefully.

**Anti-patterns to avoid (per CONTEXT.md):**
- Do NOT make multiple LLM calls per scan type -- single pass only
- Do NOT use ts-morph -- LLM-only analysis
- Do NOT auto-filter backlog items -- include all with relevance scores
- Do NOT let Brain influence effort/impact/risk scoring
