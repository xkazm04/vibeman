<task type="auto">
  <name>Task 1: Refactor executeStage.ts — domain scheduler + file verification + spec reading</name>
  <files>src/app/features/Manager/lib/conductor/stages/executeStage.ts</files>
  <action>
    Rewrite `executeExecuteStage` to replace DAG-based dispatch with domain-scheduler-based dispatch. The function signature changes: it now receives `runId` and reads specs from `specRepository` instead of receiving a `BatchDescriptor`.

    **New input interface** (replace the old `ExecuteInput`):
    ```typescript
    interface ExecuteInput {
      runId: string;
      config: BalancingConfig;
      projectId: string;
      projectPath: string;
      projectName: string;
      abortSignal?: AbortSignal;
      onTaskUpdate?: (tasks: ExecutionTaskState[]) => void;
    }
    ```

    **Core changes:**

    1. **Read specs from DB**: Replace `batch.requirementNames` with `specRepository.getSpecsByRunId(runId)`. Filter to specs with status `pending`.

    2. **Initialize SchedulerState**: Build `{ pending: specs, running: new Map(), completed: new Set(), failed: new Set() }`.

    3. **Dispatch loop** (replaces DAG loop):
       - Call `getNextBatch(schedulerState, maxParallel)` to get non-overlapping specs
       - If empty and running.size > 0, wait 2s and continue
       - If empty and running.size === 0, break (all done or blocked)
       - For each spec in batch:
         a. Update `schedulerState.running.set(spec.id, getAllPaths(spec.affectedFiles))`
         b. Remove from `schedulerState.pending`
         c. Call `specRepository.updateSpecStatus(spec.id, 'executing')` — DB update BEFORE dispatch
         d. Take file snapshots: `snapshotFiles(projectPath, [...spec.affectedFiles.modify, ...spec.affectedFiles.create])`
         e. Dispatch CLI execution

    4. **Read spec file content**: Instead of `readRequirement()`, read the spec markdown from disk:
       ```typescript
       const specDir = path.join(process.cwd(), '.conductor', 'runs', runId, 'specs');
       const filename = specFileManager.formatFilename(spec.sequenceNumber, spec.slug);
       const specContent = fs.readFileSync(path.join(specDir, filename), 'utf-8');
       ```

    5. **CLI dispatch**: Use existing `startExecution` pattern but with `useWorktree: false` (per research recommendation — domain isolation replaces worktree). Provider/model from `config.executionProvider`/`config.executionModel`.

    6. **Post-execution verification**: After CLI process completes (polling loop returns):
       - If exit code non-zero: mark failed directly (no file check needed per user decision)
       - If exit code 0: call `verifyExecution(projectPath, spec.affectedFiles, beforeSnapshots)`
         - If verification fails: mark as FAILED with verification reason
         - If verification passes: mark as completed
       - Call `specRepository.updateSpecStatus(spec.id, 'completed'|'failed')` — DB update AFTER execution
       - Update scheduler state: move spec from running to completed/failed

    7. **Keep existing patterns**: Rate limit backoff, abort signal handling, onTaskUpdate callback emissions.

    8. **Return shape** stays as `ExecuteStageResult { results: ExecutionResult[], allCompleted: boolean }`.

    Remove the DAGScheduler import and `readRequirement` import. Add imports for domainScheduler, fileVerifier, specRepository, specFileManager, and `fs`/`path`.
  </action>

  </verify>
