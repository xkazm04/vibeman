**specTemplate.ts — Spec markdown renderer:**

Export `renderSpec(data: SpecRenderData): string` that produces a structured markdown document with the 7 sections in the locked order:

1. `# {title}` (h1 heading)
2. `## Goal` — `data.goalDescription`
3. `## Acceptance Criteria` — Each criterion as `- GIVEN {given} WHEN {when} THEN {then}` (per user decision: GIVEN/WHEN/THEN format, machine-parseable)
4. `## Affected Files` — Render `data.affectedFiles` as a fenced JSON code block (`\`\`\`json ... \`\`\``)
5. `## Approach` — `data.approach` (strategy-level outline, not step-by-step)
6. `## Code Conventions` — Only rendered if `data.codeConventions` is not null. Each convention as `- **{confidence}:** {rule}` (per user decision: labeled by confidence)
7. `## Constraints` — Each constraint as `- {constraint}` (always present per user decision)
8. `## Complexity` — Just the letter: `data.complexity`

Use string interpolation (template literals) per RESEARCH.md recommendation. Join sections with double newlines.

Also export `generateSlug(title: string): string` — lowercase, replace non-alphanumeric with hyphens, trim hyphens, limit to 50 chars. Simple ASCII only per RESEARCH.md.

Also export `deriveComplexity(effort: number): SpecComplexity` — S if effort <= 3, M if effort <= 6, L if effort > 6. Maps to complexity_1/2/3 in model routing (S=complexity_1, M=complexity_2, L=complexity_3).

**fileDiscovery.ts — ts-morph affected file analysis:**

Export `discoverAffectedFiles(projectPath: string, entryFiles: string[], maxDepth?: number): AffectedFiles`:

1. Create a ts-morph `Project` with `skipAddingFilesFromTsConfig: true` (per RESEARCH.md pitfall 1 — performance)
2. Attempt to load tsconfig from `path.join(projectPath, 'tsconfig.json')` if it exists, otherwise create without it
3. Add each entry file via `project.addSourceFileAtPathIfExists()`
4. For each added source file, traverse import declarations up to `maxDepth` (default 2):
   - Track visited files in a Set to prevent circular loops (per RESEARCH.md pitfall 2)
   - Resolve import paths to absolute paths
   - Classify as `modify` (file exists on disk) or skip if not resolvable
5. Files in `entryFiles` that exist on disk go into `modify`, those that don't exist go into `create`
6. `delete` array is always empty (spec writer doesn't determine deletions — that comes from the backlog item description if needed)
7. Return `{ create, modify, delete: [] }` with paths relative to `projectPath`

Also export `validateAffectedFiles(affectedFiles: AffectedFiles, projectPath: string): { valid: boolean; errors: string[] }`:
- Files in `modify` must exist on disk (fs.existsSync)
- Files in `delete` must exist on disk
- Files in `create` must NOT exist on disk
- Per user decision: exact validation at spec time

Create the spec writer stage function that orchestrates all spec modules.

Export `async function executeSpecWriterStage(input: SpecWriterInput): Promise<SpecWriterOutput>`:

1. **Create spec directory:** Call `specFileManager.ensureSpecDir(input.runId)`

2. **For each approved item** (ordered by array index = priority order per user decision):
   a. **File discovery:** Call `discoverAffectedFiles(input.projectPath, item.filePaths)` to get `AffectedFiles`. If `item.filePaths` is empty, produce `{ create: [], modify: [], delete: [] }` rather than calling ts-morph with no entries.
   b. **Validate files:** Call `validateAffectedFiles(affectedFiles, input.projectPath)`. Log warnings for validation errors but don't fail — the spec is still useful with imperfect file claims.
   c. **Derive complexity:** Call `deriveComplexity(item.effort)`.
   d. **Query Brain:** Call `getBehavioralContext(input.projectId)`. Extract code conventions by filtering `topInsights` for relevance. Per user decision: query Brain per-spec (not batched), but since `getBehavioralContext` returns project-level data, call it once and filter per spec. If `ctx.hasData === false`, set `codeConventions = null` (per user decision: omit Code Conventions section entirely).
   e. **Extract conventions:** For each `topInsight`, create a `CodeConvention` with `confidence >= 80` = "Strong pattern", else "Emerging pattern". Include `insight.title` as `source`. Cap at 5 conventions per spec.
   f. **Build render data:** Construct `SpecRenderData` with:
      - `title`: item.title
      - `goalDescription`: `input.goalContext.description` + context about this specific item
      - `acceptanceCriteria`: Derive 3-5 GIVEN/WHEN/THEN assertions from item.description. Since this is an automated pipeline (no LLM call in spec writer itself), generate structural assertions based on affected files: e.g., "GIVEN the implementation is complete WHEN TypeScript compiles THEN no errors in {affected files}". The real behavioral criteria come from the backlog item description — parse any existing assertions or create file-existence/compilation assertions.
      - `affectedFiles`: from step (a)
      - `approach`: Extract strategy from item.description. If description is short, use `"Implement {item.title} by modifying the affected files listed above. Follow existing codebase patterns."`
      - `codeConventions`: from step (e) or null
      - `constraints`: Always include at least: "Do NOT modify files outside the affected files list", "Do NOT add new dependencies without explicit justification". Add category-specific constraints based on `item.category`.
      - `complexity`: from step (c)
   g. **Render spec:** Call `renderSpec(renderData)` to get markdown string.
   h. **Generate filename:** `specFileManager.formatFilename(index + 1, generateSlug(item.title))`
   i. **Write file:** `specFileManager.writeSpec(specDir, filename, markdown)`
   j. **Persist metadata:** `specRepository.createSpec({ id: uuidv4(), runId, backlogItemId: item.id, sequenceNumber: index + 1, title: item.title, slug: generateSlug(item.title), affectedFiles, complexity })`

3. **Return:** `{ specs: allSpecMetadata, specDir }`
